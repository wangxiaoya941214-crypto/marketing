## 3. 已记录问题

## 01. 依赖升级后存在兼容风险

### 问题现象

项目依赖版本不是最新主线，升级后如果只改版本号，不处理兼容问题，容易导致：

- 构建失败
- 类型报错
- 路由异常
- 启动失败

### 根因

这次升级涉及：

- `Vite 6 -> 8`
- `TypeScript 5.8 -> 6`
- `Express 4 -> 5`

这些都不是纯 patch 升级，而是会影响运行和类型系统的升级。

### 修复动作

已完成的修复：

1. 升级主依赖版本
2. 调整 Express 生产环境路由回退
3. 调整服务启动逻辑
4. 增加 `src/vite-env.d.ts`
5. 显式收紧 `tsconfig.json` 的 `include / exclude`
6. 补齐 React 类型包
7. 更新 README 里的 Node 版本要求

### 回归检查

已执行：

- `npm run lint`
- `npm run build`
- 本地启动烟雾测试
- `curl` 检查服务返回 `200 OK`

### 防复发规则

以后只要出现以下任一情况，就必须按“兼容升级”处理，不能只改版本号：

- 构建器 major 升级
- TypeScript major 升级
- 服务框架 major 升级

每次升级后必须执行：

```bash
npm run lint
npm run build
```

并至少确认一次：

- 页面能打开
- API 能响应
- 路由没有失效

## 02. 结果页存在“后端有、前端没展示”的能力丢失

### 问题现象

后端分析引擎已经返回了完整结果，但前端结果页一开始只展示了部分内容。  
用户只能看到简化版看板，很多已经算出来的建议和报告没有露出。

### 根因

前端结果页最初只渲染了：

- KPI 卡片
- 预算占比 vs 成交占比
- 分产品漏斗

但没有接入这些已存在字段：

- `analysis`
- `contentRanking`
- `budgetRecommendations`
- `actionPlan`
- `scalePlan`
- `reliability`

### 修复动作

已新增并接入：

1. 模块导航栏
2. 可靠性说明
3. 完整分析报告
4. 内容排名
5. 预算建议
6. 动作清单
7. 放量建议

实现方式：

- 抽到独立组件文件
- 在结果页中增量接入
- 不改原有已展示模块逻辑

### 回归检查

已执行：

- `npm run lint`
- `npm run build`

并确认：

- 新模块已接入结果页
- 旧模块未被破坏
- 结果页导航可跳转

### 防复发规则

以后只要 `MarketingDashboardData` 新增业务字段，就必须同步检查：

1. 前端是否已有展示
2. 如果不展示，是否在 PRD 或代码信息报告里注明原因

不能再出现“后端有能力，但前端长期没接”的情况。

## 03. 上期数据能力在引擎里有基础，但前端没有入口

### 问题现象

分析引擎原本已经支持一部分上期数据字段，但用户在匹配页无法输入，导致环比分析能力无法真正使用。

### 根因

后端数据结构里存在 `previous`，且 `metricsTable` 已支持 `mom` 字段。  
但前端匹配页最初没有“上期数据”区块，用户没有录入口。

### 修复动作

已新增：

1. 匹配页“上期数据（选填，用于环比分析）”折叠区
2. 4 个输入项：
   - 上期成交量
   - 上期 CPS
   - 上期整体转化率
   - 上期总花费
3. 结果页“环比变化”模块

### 回归检查

已执行：

- `npm run lint`
- `npm run build`

并确认：

- 匹配页能录入上期数据
- 结果页会根据上期数据展示环比变化
- 不填上期数据时不影响正常分析

### 防复发规则

以后新增任何“趋势 / 环比 / 对比”能力时，必须先检查：

```text
前端有没有输入入口？
后端有没有接收字段？
分析引擎有没有消费？
结果页有没有展示？
```

这四层少一层，都算链路不完整。

## 04. 上期数据口径不完整，导致前端需求与引擎结构不匹配

### 问题现象

产品需要的是聚合口径的：

- 上期成交量
- 上期 CPS

但原始引擎结构更偏分产品口径，前端需求没法直接无损落地。

### 根因

`previous` 里原本只有：

- `flexibleDeals`
- `superDeals`
- `flexibleCps`
- `superCps`
- `cpl`
- `overallConversionRate`
- `totalSpend`

缺少聚合口径的：

- `totalDeals`
- `overallCps`

### 修复动作

已在 `previous` 中补齐：

- `totalDeals`
- `overallCps`

同时补齐了：

- CSV 模板字段
- CSV 解析逻辑
- 输入标准化逻辑
- 结果页环比展示逻辑

### 回归检查

已确认：

- `MarketingInput.previous` 结构已扩展
- 上期数据输入能进入引擎
- 环比模块能正确消费聚合口径

### 防复发规则

以后新增字段时，不能只看“是否能勉强复用老字段”，必须先判断：

- 这个字段是不是一个独立业务口径？
- 如果是，就要单独建字段，而不是借道假装复用

否则会导致：

- 口径歧义
- 前端展示错误
- 后续分析逻辑越来越绕

## 05. TypeScript 编译范围不清，构建产物被误扫进类型检查

### 问题现象

升级之后，`tsc` 会把 `dist/` 下的构建产物也当成源码处理，导致类型检查报错。

### 根因

`tsconfig.json` 最初没有显式限制 `include / exclude`，默认模式下把构建产物也纳入了扫描范围。

### 修复动作

已在 `tsconfig.json` 中显式增加：

- `include`
- `exclude`

确保只扫描：

- `src`
- `shared`
- `server.ts`
- `vite.config.ts`

排除：

- `dist`
- `node_modules`

### 回归检查

已执行：

- `npm run lint`

确认：

- 不再出现 `dist` 相关类型报错

### 防复发规则

以后如果新增新的源码目录，必须同步更新 `tsconfig.json`。  
不能依赖默认 include 规则。

## 06. 结果模块新增时，必须采用“增量接入”而不是重写旧逻辑

### 问题现象

结果页需要补大量模块，如果直接在原来的结果区里混写，很容易引发：

- 旧模块回归
- 逻辑耦合
- PDF 导出异常
- 代码可读性下降

### 根因

结果页原本已经承担：

- KPI 看板
- 图表
- 漏斗
- 导出

如果继续在同文件内横向堆逻辑，维护成本会快速上升。

### 修复动作

本轮采取的方式是：

1. 把新增结果模块抽到 `src/components/result-modules.tsx`
2. 把上期数据与环比逻辑抽到 `src/components/previous-period.tsx`
3. 在 `App.tsx` 中只做挂接

### 回归检查

已确认：

- 旧结果页模块保持可用
- 新模块成功接入
- 构建通过

### 防复发规则

以后新增结果模块时，优先采用：

```text
新增组件文件
  -> App.tsx 中接入
  -> 避免在旧模块内部直接堆新逻辑
```

除非是局部文案调整，否则尽量不要在旧模块里混入大段新逻辑。

## 08. 缺少错误记录会导致同类问题反复出现

### 问题现象

如果只修代码、不记录错误类型和修复方式，后续很容易再次出现：

- 前后端链路断裂
- 字段口径不一致
- 升级后回归
- 结果页漏接字段

### 根因

项目原本没有专门记录“错误现象 / 根因 / 修复动作 / 回归检查 / 防复发规则”的地方。

### 修复动作

已建立本文件，并统一规定后续错误记录格式：

```md
### 问题现象
### 根因
### 修复动作
### 回归检查
### 防复发规则
```

### 回归检查

已确认：

- 本文件已创建
- 文档导航已引用本文件

### 防复发规则

以后任何真实问题，只要进入“修复”阶段，就必须补一条记录。  
不要等问题累计到很多次之后再回忆。
