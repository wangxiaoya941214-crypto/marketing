# 背景知识文档

## 1. 文档用途

本文件记录当前产品背后的业务上下文、术语解释、分析口径和品牌限制。  
它的目标不是替代代码，而是帮助产品、设计、研发、Prompt 调优时对齐同一套业务语言。

---

## 2. 业务背景

当前产品服务于 `SUPEREV 超级电动` 的营销分析与复盘场景。

它关注的不是泛营销，而是更偏：

- 线索获取
- 私域承接
- 高意向转化
- 最终成交

也就是说，它围绕的是一条相对明确的成交链路，而不是纯曝光类内容分析工具。

---

## 3. 当前产品里的两类业务对象

## 3.1 灵活订阅（flexible）

从代码里的定义看，灵活订阅更偏：

- 短期需求
- 先试后买
- 阶段性过渡
- 低试错成本

用户关心点：

- 随时可停
- 决策压力低
- 快速知道是否适合自己

## 3.2 超级订阅（super）

从代码里的定义看，超级订阅更偏：

- 长期用车
- 更稳定的方案
- 更重视全新车与长期履约

用户关心点：

- 总账是否划算
- 长期承诺是否值得
- 交付与履约是否稳定

---

## 4. 核心术语解释

| 术语 | 含义 |
|---|---|
| 留资 / 客资 / 线索 | 用户留下联系方式后的有效线索 |
| 转私域 | 用户从公域进入微信等私域承接链路 |
| 高意向 | 已经进入明确考虑、具备成交可能的用户 |
| 成交 | 最终下单或形成明确成交结果 |
| 漏斗 | 用户从留资到成交的分阶段路径 |
| CPL | 每条客资成本，`spend / leads` |
| CPS | 每次成交成本，`spend / deals` |
| CPS 红线 | 可接受的每次成交成本上限 |
| 目标完成率 | 实际成交 / 目标成交 |
| 权重占比 | 某产品花费占整体花费的比例，或某产品成交占整体成交的比例 |
| 质量分 | 内容成交贡献占比 / 线索占比 的近似指标 |

---

## 5. 当前分析口径

## 5.1 漏斗阶段

当前固定为 4 段：

1. 留下联系方式
2. 加微信 / 进私域
3. 明确有意向
4. 最终成交

## 5.2 核心公式

- `CPL = spend / leads`
- `CPS = spend / deals`
- `成交目标完成率 = deals / targetDeals`
- `留资 -> 私域转化率 = privateDomain / leads`
- `私域 -> 高意向转化率 = highIntent / privateDomain`
- `高意向 -> 成交转化率 = deals / highIntent`
- `整体转化率 = deals / leads`

## 5.3 内容评价逻辑

当前内容评价主要围绕：

- 带来多少留资
- 每条留资花多少钱
- 成交贡献是否匹配

它更像“效果诊断排名”，不是内容创意理解引擎。

---

## 6. 当前不具备的口径

以下能力在代码里还没有完整实现：

- ROI / ROAS
- 渠道横向对比
- 销售个人转化率
- 历史趋势分析
- 自动预警规则中心

所以后续写需求时，不能把这些默认成“现有能力”。

---

## 7. 品牌与表达红线

当前代码里已经明确写入了内容红线，后续无论是 Prompt、分析、UI 文案都应遵守：

1. 不能直接和 4S 店比价
2. 不能直接和官方渠道比价
3. 不能承诺超出真实交付能力
4. 不能传播“明日车库”
5. 不能混用灵活订阅和超级订阅卖点

这些限制已经影响：

- 审计规则
- AI 报告 Prompt
- 建议文案边界

---

## 8. 当前建议背后的业务假设

当前引擎里的建议，大致基于这些假设：

### 8.1 如果预算占比高，但成交占比低

说明该产品/方向花钱效率差，预算应该缩。

### 8.2 如果某内容 CPL 低，或成交贡献优于线索贡献

说明这条内容更值得加量。

### 8.3 如果最大流失环节和最差转化环节不是同一步

优先救“流失人数最多”的环节。

### 8.4 如果整体评级为绿

才进入“放量建议”阶段。

这些都属于规则型业务判断，不是强因果证明。

## 9. 当前背景知识缺口

目前项目里还缺少这些正式背景资料：

1. 渠道维度定义  
   例如抖音、小红书、品牌号、私域的正式口径

2. 成交口径定义  
   什么算成交、什么算高意向、谁负责确认

3. 内容板块分类标准  
   `board` 目前有字段，但缺少标准分类字典

4. 目标值制定规则  
   为什么某产品的目标与红线是当前数值

5. 真实业务案例  
   当前建议里常提“案例”，但仓库中没有正式案例库

## 11. 面向后续迭代的背景建议

如果后续要新增模块，建议优先补齐这些背景知识：

### 新增“渠道对比漏斗”前

要先补：

- 渠道枚举
- 渠道字段采集口径
- 各渠道是否都能拿到漏斗数据

### 新增“动作追踪”前

要先补：

- 动作责任人体系
- 跟踪周期
- 验证指标标准

### 新增“异常预警”前

要先补：

- 预警阈值定义
- 哪些预警需要打断用户
- 哪些预警只是提示

## 12. 一句话总结

这个产品当前已经具备“营销诊断”的基础业务上下文，但要往“经营分析平台”走，还需要补齐渠道、趋势、动作追踪和品牌/业务口径资料。
